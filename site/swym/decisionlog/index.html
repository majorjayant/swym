Decision Log

Written as it happened, timestamped from a 90 minute clock. First 25 mins went into setup — prompt design on Gemini & Opus 4.8, then git & cloudflare pipeline on Opus 4.7, live before I even opened the datasets. Every entry here is my call, my words. The AI transcribed & computed, it was never allowed to choose the problem.

[T+25] Clock start — what I looked at first

Phase 1 was facts only: "No recommendations, no ideas, no problem framing." One profiling script covering structure, field inventory (partial fields matter more than complete ones), cross-file overlap, key analysis, integrity, session-field stress tests, distributions, and the temporal relationship between the files — every number from executed code, saved to analysis/profile_output.txt.

What it told me: file A is 470 interaction events with no library_card_id and no device_session_id — the join fields the brief describes don't exist in the data. File B is 95 identified sessions, exactly one per patron. The only shared fields are branch_id and branch_terminal_id. The connection the brief implies had to be earned, not joined.

[T+33] Decision reconsidered (1): the join I almost wrote off

The agent's first pass concluded "no join is possible" because zero A events fall inside any B session window. I caught that this tested the wrong relation — its own finding showed all 120 terminal-bearing A events land 65s–1857s before the nearest same-terminal login, strictly one-sided. My rework order: "You found a pattern and then tested for a different one. Run the test your own finding points at."

Believed before: the files might not connect at all. Believe now: a deterministic pre-login bridge exists — at a 45-minute window all 120 events resolve to exactly one session each, zero ambiguity, zero conflicts (stress_and_bridge_output.txt). I still chose not to build on it — as a decision, not a data failure. The same rework killed terminal_session_id as a "session": 86/150 groups span multiple branches, median span ≈16 days; any per-visit funnel died with it.

[T+62] Decision reconsidered (2): option 5's "high reach"

I called the high-holds flag "impactful and high reach." The agent flagged the computed number: 8 patrons of 95 = 8.4% — low reach, high per-account impact. I accepted the correction; RICE used the measured reach and option 5 dropped to last among buildables. It stayed deferred.

[T+61] The problem I decided to solve

The first thing that I wanted to solve after seeing the assignment and data was any kind of engagement issue or visibility issue. Once I went deeper I saw staff didn't know which titles had long queues, a very genuine issue that had to be taken up. And the titles that never get seen, that's the kind of reach issue which I want to target during my initial build planning. The export issue was highlighted after phase 1 forensics itself — I wanted to solve that too, as a recommendation.

So: go with 8, 2 & 3 — basic, performance & delightful, using the Kano model. 8 (data contract) is the basic fix the source system needs, recommended not built; 2 (hold-queue pressure) is the performance feature staff would feel; 3 (interest-vs-circulation gap) is the delighter, shipped with its confidence stated on every row.

Why over the others, naming them: 1 (pre-login linker) not needed for now — agents/bots can handle it later, network controls exist. 4 (terminal utilisation) low user impact, compliance-side, later if resources allow. 5 (high-holds flag) usable but low confidence, and reach turned out to be 8/95. 6 (session funnel) do-not-build — the field it needs is provably not a session. 7 (patron profiles) later, once the system is capable enough.

The agent argued once against the three-way split (three partials vs one finished; option 3's 22-checkout evidence base; option 5 given up as the only completable build) — recorded in SCRATCH.md. I kept all three.
